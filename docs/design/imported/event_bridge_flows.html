Event Bridge 흐름 명세 — 시작과 끝이 명확한 시나리오

작성: 2026-04-22 · 기준 코드: backend/management/services/event_bridge.py, backend/management/server.py:240 ReportHandoffAck, backend/management/services/amr_state_machine.py:146 confirm_handoff
목적: 기존 event_bridge_design.html 의 혼합 관계도가 화살표 교차로 시작과 끝이 불분명했던 문제를 해결. 각 시나리오를 독립된 세로 타임라인으로 분리.

목차 — 3가지 실제 시나리오

  1. 시나리오 1 — 관리자 승인 → 주조 task 자동 생성 → 장비 배정 (설계 · Phase 2 이벤트 기반)
  2. 시나리오 2 — 공정 완료 → 다음 공정 자동 전이 (설계 · Phase 2 이벤트 기반)
  3. 시나리오 3 — 후처리 버튼 → handoff 완료 (현재 실제 구현·검증 완료)
  4. 공통 원칙 4가지
  5. 읽는 법 (범례)

시나리오 1 — 관리자 승인 → 주조 task 자동 생성 → 장비 배정

① PyQt 승인 버튼 → 로봇팔 주조 시작까지
관리자가 승인 버튼을 누른 후, 서버가 자동으로 task 를 만들고 장비에 배정하는 전체 흐름. Event Bridge 가 두 번 개입한다 (ORDER_APPROVED → TASK_CREATED 각 1회). 설계 단계 — Phase 2 이벤트 기반 도입 시 활성화 예정.
시작
관리자가 PyQt "생산 시작" 버튼을 누름

Event Bridge 구독 테이블 조회 → TASK_CREATED 구독자: [TaskAllocator]

task 가 PROC 상태로 DB 저장, 로봇팔 RA1 에 배정됨
DB 상태: ord.ord_stat = APPR · equip_task_txn(txn_id=100).txn_stat = PROC · equip_stat(RA1) = BUSY
다음: RobotExecutor 가 실제 장비 명령 전송 (시나리오 2 로 연결)

이 시나리오에서 Event Bridge 가 한 일 / 안 한 일

  • ✅ 이벤트 2회 라우팅 (ORDER_APPROVED, TASK_CREATED)
  • ✅ 구독자 매칭 → 핸들러 동기 호출
  • ❌ DB 접근 0회 — 모든 DB write 는 State Manager 가 담당
  • ❌ 비즈니스 로직 판단 0회 — "승인되면 CAST 를 만들어야 한다" 는 TaskManager 가 결정

시나리오 2 — 공정 완료 → 다음 공정 자동 전이

② 로봇팔 CAST 완료 → AMR 이송 task 자동 생성
"Executor 만이 이벤트를 발행한다" 는 고정 규칙이 가장 잘 드러나는 흐름. 공정 완료 신호가 다음 공정 연쇄 실행의 출발점이다. 설계 단계 — Phase 2 이벤트 기반 도입 시 활성화 예정.

이 시나리오의 핵심

  • Event Bridge 가 두 번 개입 (TASK_COMPLETED → TASK_CREATED)
  • TaskManager 가 "다음 공정 판단" 의 주인 — Bridge 는 판단 안 함, 그저 라우팅
  • WebSocket 도 같은 이벤트(TASK_COMPLETED) 를 받음 — 1:N 브로드캐스트의 진가

공통 원칙 4가지

원칙 1 — 발행 주체는 "액션 수행자"

  • 이벤트는 상태를 실제로 바꾼 컴포넌트 가 발행한다
  • API Handler · TaskManager · RobotExecutor · 외부 장비(ESP32) 가 발행 주체
  • Event Bridge 자신은 절대 publish 호출하지 않는다 (순수 라우터)
  • State Manager 도 기본적으로 발행하지 않음. 단 STATE_CHANGED 이벤트만 예외적으로 허용 가능

원칙 2 — SSOT 는 State Manager 단독

  • DB CRUD 는 오직 State Manager 가 수행
  • Event Bridge 는 DB 에 접근하지 않음 (read/write 둘 다)
  • 다른 컴포넌트가 DB 조작이 필요하면 반드시 State Manager 경유

원칙 3 — 이벤트 vs 직접 호출 판단 기준

  • 응답이 필요하면 → 직접 호출 (Bridge 는 fire-and-forget)
  • 수신자가 1명 + 응답 필요 → 직접 호출
  • 트랜잭션 원자성 필요 → 직접 호출
  • N명에게 상태 변화 전파 → Bridge publish
  • 공정 체인 자동 연결 → Bridge publish

원칙 4 — 한 함수 안에 direct call + publish 공존 정상

한 API Handler 안에서 (a) DB write 는 State Manager 직접 호출, (b) 상태 변화 알림은 Bridge publish 로 나눠서 하는 게 정상 패턴. "모든 것을 Bridge 로" 는 안티패턴.
