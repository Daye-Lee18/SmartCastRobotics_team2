읽는 순서 (위 → 아래로 한 방향)

  1. 🟢 시작 (0) — 외부 트리거: PyQt 버튼·ESP32 버튼·장비 완료 신호 중 하나가 발생
  2. 입력 채널 (1) — gRPC / Serial / ROS2 경유로 서버로 들어옴
  3. 발행 주체 (2) — API Handler · Robot Executor · Task Manager 중 하나가 액션을 수행한 후 publish()
  4. 🟠 Event Bridge (3) — 이벤트 타입 매칭 → 등록된 구독자 리스트 조회 → 호출 준비
  5. 구독자 (4) — Task Manager / Task Allocator / AMR FSM / WebSocket 등이 on_event() 호출 받음
  6. 🔴 State Manager (5) — 모든 DB 조작이 여기 한 곳으로 수렴 (SSOT)
  7. 🔴 끝 (6) — PostgreSQL — 상태 영구 저장. 한 사이클 완료.

화살표는 전부 위에서 아래로만 흐른다. 읽기 전용 조회(점선 파랑)만 예외적으로 옆으로 뻗는다.

4가지 원칙 (다이어그램에서 직접 확인 가능)

원칙 1 — Executor 만 이벤트 발행 (Confluence 41648180 고정 규칙)

  • 중앙 "Robot Executor" 박스에 ★ 이벤트 발행 주체 표시
  • API Handler · Task Manager 도 각자 액션 완료 후 발행 가능 (액션 수행자 원칙)
  • Event Bridge 자신은 절대 publish 호출하지 않음 (순수 라우터)

원칙 2 — Event Bridge 는 수신·라우팅·호출만. SSOT 는 State Manager 단독

  • Bridge 박스에서 DB 로 향하는 화살표 없음 (read-only meta 점선만 State Manager 경유)
  • 모든 DB write 화살표는 구독자 → State Manager → PostgreSQL

원칙 3 — DB 쓰기는 State Manager 만. Bridge 의 DB 직접 접근은 SSOT 원칙 위반

  • 빨강 화살표는 모두 State Manager 로 수렴
  • State Manager → PostgreSQL 이 유일한 DB write 경로

원칙 4 — Bridge 가 필요한 정보는 이벤트 페이로드 또는 State Manager 읽기 전용 래퍼로

  • Bridge → State Manager 로 향하는 점선 파란색 화살표가 유일한 조회 경로
  • 이조차 읽기 전용. write 는 불가
